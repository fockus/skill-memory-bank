# Proposal — Architect / Decomposition-as-Artifact stage перед параллельным исполнением

**Date:** 2026-06-16
**Backlog:** I-080 (`backlog.md`)
**Status:** PROPOSAL (pre-spec — основа для будущей `specs/<topic>/`)
**Author:** governed-flow session (dynamic-flow Phase 2/3 контекст)
**Related:** `specs/parallel-pipeline/`, `specs/parallel-team-execution/`, `specs/dynamic-flow/` (Task 9–13 fanout/fence), `agents/mb-architect.md`, `scripts/mb-fanout.sh`, паттерн `fanout-synthesize` (`flow-templates/`)

---

## 1. Одной строкой

Промотировать декомпозицию фичи из эфемерной «в голове оркестратора» работы в **явный артефакт — execution-DAG с зависимостями и владением файлами**, который проходит **один** ревью-гейт до фан-аута; исполнители идут по фиксированным role-контрактам, каждый проходит per-item verify, а совместный integration-reviewer (barrier) → судья выносит GO/NO_GO. Включается только выше порога сложности, **off by default**.

## 2. Проблема / мотивация

Сегодня раскладку фичи на параллельные потоки делает сам оркестратор (main-loop LLM):

1. **Декомпозиция не аудируется и не воспроизводится.** Она живёт в контексте оркестратора, не сохраняется как артефакт, не реплеится, не резюмится после компакта. Это нарушает принцип, который уже зашит в `tasks.md` (`<!-- mb-task:N -->`): декомпозиция должна быть durable-артефактом.
2. **На сам план нет гейта.** Governed-workflow (`implement → verify → review → judge`) проверяет *реализацию*. Плохую декомпозицию — неверные границы задач, пропущенные задачи, скрытые зависимости, поданные как параллельные, — никто независимо не валидирует. А цена плохого split умножается на число параллельных исполнителей: это самая high-leverage точка для гейта, и сейчас она пустая.
3. **Риск гонки при параллелизме.** Опасна не раздача, а ошибка в графе зависимостей: запустить параллельно две задачи, которые правят один файл, или где B зависит от интерфейса, который ещё создаёт A. Без явной декларации владения файлами fan-out небезопасен (см. §6 — прямая связка с fence из dynamic-flow Task 12).

## 3. Goals / Non-goals

**Goals**
- G1. Декомпозиция = durable, ревьюируемый, реплеящийся артефакт (execution-DAG).
- G2. Один независимый гейт на корректность DAG **до** траты токенов исполнителей.
- G3. Явная декларация per-task: зависимости + владение файлами (paths) + контракт (DoD/Testing/Covers).
- G4. Безопасный параллелизм: непересекающееся владение файлами как инвариант, проверяемый детерминированно.
- G5. Threshold-gating: тяжёлый путь только для крупных фич; маленькие фичи — сегодняшний плоский поток.

**Non-goals**
- N1. **НЕ** второй механизм декомпозиции рядом с `/mb sdd`/`planner`. Это *расширение* существующего артефакта, а не дубль.
- N2. **НЕ** архитектор, пишущий прозу-промпты исполнителям (см. §4.3 — coupling-антипаттерн).
- N3. **НЕ** менять дефолт. Design-контракт проекта: «expensive paths off by default; defaults never change without explicit opt-in».
- N4. **НЕ** заменять судью или per-item verify — они уже есть в `/mb work`.

## 4. Предлагаемый дизайн

### 4.1 Архитектор-стадия → execution-DAG (артефакт)

Архитектор (агент `mb-architect`, опционально усиленный judge-панелью подходов) разбирает requirements и выдаёт **execution-DAG**, расширяющий существующий `tasks.md`/новый sidecar (напр. `specs/<topic>/dag.json` или дополнительные поля во frontmatter task-блоков):

Per-task метаданные:
- `id`, `title`
- `deps: [task-id...]` — топологический порядок; пустой = корень волны параллелизма
- `owns_paths: [glob/path...]` — файлы/области, которыми задача ВЛАДЕЕТ эксклюзивно в своей волне
- `contract` — DoD (SMART), Testing-требования, `Covers: REQ-...`
- `role` — какой role-агент исполняет (`mb-backend`/`mb-frontend`/…)
- `wave` — производное: уровень в топосортировке (задачи одной волны без общих `owns_paths` идут параллельно)

### 4.2 Один ревью-гейт на DAG (дёшево)

До фан-аута — **одно** независимое ревью именно DAG (не полный governed-цикл вокруг планирования). Проверяет: полноту покрытия REQ, корректность зависимостей (нет циклов, нет «ложного параллелизма»), **непересечение `owns_paths` внутри волны** (детерминированно, скриптом), вменяемость границ задач (нет over-split / under-split). Выносит ACCEPT / REWORK на сам план.

### 4.3 Исполнители — по фиксированным role-контрактам (Contract-First)

Архитектор выдаёт **структурный контракт задачи**, а не прозу-промпт. «Как» подставляет фиксированный role-промпт исполнителя (`mb-backend.md` и т.д.). Это:
- наше CRITICAL-правило Contract-First (Protocol/ABC → contract tests → implementation);
- развязывает архитектора и внутренности исполнителей (иначе при смене role-агента ломаются все промпты);
- так уже работает `/mb work` (task-блоки несут Covers/DoD/Testing, role-агент даёт реализацию).

### 4.4 Per-item verify + совместный integration-reviewer (barrier) → судья

- **Per-executor verify** (уже есть в `/mb work`): задача проходит свой DoD локально, рано отсекает локальные дефекты.
- **Совместный integration-reviewer** — **barrier**: «состыковались ли интерфейсы, нет ли двойной реализации, цельна ли интеграция». Это структурно видно ТОЛЬКО при наличии всех результатов волны (cross-item контекст) → законный barrier (в отличие от дефолтного правила «по умолчанию pipeline»).
- **Судья** (`mb-judge`) — терминатор: GO / GO_WITH_BACKLOG / NO_GO. (См. [[judge-terminates-review-loop]].)

### 4.5 Threshold-gating

Тяжёлый путь активируется только выше порога сложности: N задач ≥ порога, multi-file, или оценка стоимости. Маленькие фичи (2–3 задачи) остаются на плоском потоке — overhead архитектора+DAG-ревью иначе съест выигрыш от параллелизма. Off by default, opt-in флагом/`pipeline.yaml`.

## 5. Как встраивается в существующую систему (не дубль)

| Существует | Роль в предложении |
| --- | --- |
| `/mb sdd` + `planner` | requirements→design→tasks — расширяем его выходом DAG (deps + owns_paths), а не создаём новый механизм |
| `/mb work` | per-item implement→verify→review→judge — переиспользуем как исполнительный движок |
| `scripts/mb-fanout.sh` | примитив фан-аута — runtime для волн DAG |
| паттерн `fanout-synthesize` | runtime-аналог; DAG — это его **планировочная** половина |
| `agents/mb-architect.md` | агент-исполнитель архитектор-стадии |
| `specs/parallel-pipeline/`, `specs/parallel-team-execution/` | предшествующие специи параллелизма — переиспользовать инварианты, проверить на конфликт |

Вывод: предложение **достраивает недостающую планировочную половину** к уже существующему runtime фан-аута, а не вводит параллельный механизм.

## 6. Прямая связка с dynamic-flow Task 12 (fence)

Task 12 строит fence-discipline: параллельные ветки пишут `.mb-flow/branch-<i>.json`, фенс пишется один раз серийно инициатором, без гонки. Это **нижняя** (runtime) половина гарантии. `owns_paths` из DAG — **верхняя** (планировочная) половина: декларация эксклюзивного владения файлами на волну. Вместе они дают полную гарантию безопасного параллелизма: архитектор гарантирует, что ветки не пересекаются по файлам; fence гарантирует, что их результаты сливаются без гонки. Без `owns_paths` fence защищает только запись состояния, но не сам конфликт правок.

## 7. Open questions

- OQ1. Формат DAG: расширять frontmatter `tasks.md` или sidecar `dag.json`? (sidecar проще валидировать скриптом; frontmatter ближе к существующему artifact-флоу.)
- OQ2. Гранулярность `owns_paths`: file-level или directory-level? Как трактовать shared-файлы (напр. `index.ts` public API)?
- OQ3. Кто детектит цикл/ложный параллелизм — детерминированный скрипт (`mb-dag-validate.sh`?) или ревью-агент? (Предпочтительно скрипт для топологии + агент для семантики границ.)
- OQ4. Порог threshold-gating: по числу задач, по multi-file, по оценке токенов — или композит?
- OQ5. Re-decomposition на REWORK: судья/ревью DAG возвращает план — как версионировать DAG, чтобы не потерять историю?

## 8. Черновик acceptance / EARS-seeds (для будущей спеки)

- WHEN фича помечена выше порога сложности, the system SHALL произвести execution-DAG-артефакт до запуска исполнителей.
- The execution-DAG SHALL для каждой задачи декларировать `deps`, `owns_paths`, `contract (DoD/Testing/Covers)`, `role`.
- WHERE две задачи в одной волне, the system SHALL гарантировать пустое пересечение их `owns_paths` (детерминированная проверка).
- WHEN DAG не проходит ревью-гейт, the system SHALL вернуть REWORK и НЕ запускать исполнителей.
- The system SHALL по умолчанию НЕ активировать архитектор-стадию для фич ниже порога (off by default).
- WHEN все задачи волны завершены, the system SHALL запустить integration-reviewer как barrier до судьи.

## 9. ICE (грубо)

- **Impact:** HIGH — закрывает дыру «нет гейта на план» + делает параллелизм безопасным by construction; high-leverage (ошибка плана × N исполнителей).
- **Confidence:** MED — большая часть инфраструктуры уже есть (sdd/work/fanout/fence); риск в формате DAG и threshold-калибровке.
- **Ease:** MED — это надстройка над существующими артефактами, но затрагивает sdd-генерацию, новый валидатор DAG, и wiring в `/mb work`.
- **Приоритет:** кандидат в **Phase 3** dynamic-flow (после закрытия fanout/sub-invoke Task 12–14), не мид-фазный болт.

## 10. Риски

- R1. **Раздувание пайплайна.** Митигировать: один гейт (не три), off by default, structured contracts (не проза).
- R2. **Дубль с sdd/planner.** Митигировать: расширять артефакт, а не плодить механизм (N1).
- R3. **Латентность/токены на мелочи.** Митигировать: threshold-gating (§4.5).
- R4. **Неверная калибровка порога** → либо overhead, либо плоский поток на сложной фиче. Требует данных реального использования (как ADR-009 про benchmarks).
